# 안티스푸핑(위조 얼굴 탐지) 프로젝트 — 쉬운 설명서

이 문서는 **이 분야를 처음 보는 팀원**을 위한 거예요. 전문 용어 없이, 우리가 뭘 만들고 있고 지금 어디까지 왔는지 설명합니다.

---

## 1. 이 프로젝트가 뭘 하나요?

얼굴인식 **출입통제기**(문 앞에서 얼굴 보고 열어주는 기계)에 들어갈 기능이에요.

문제: 누군가 **내 얼굴 사진**이나 **휴대폰 화면 속 내 영상**을 카메라에 들이대면, 단순한 얼굴인식기는 "본인이네" 하고 문을 열어줄 수 있어요. 이걸 막아야 해요.

그래서 우리 모델은 **"지금 카메라 앞에 있는 게 진짜 사람 얼굴이냐, 아니면 위조냐"**를 판별해요. 이걸 업계에서 **라이브니스(liveness, 생체 여부) 판별** 또는 **안티스푸핑(anti-spoofing, 위조 방지)**이라고 불러요.

판별하는 종류는 5가지예요:
- **live** = 진짜 사람 얼굴 (통과시켜야 함)
- **print** = 종이에 인쇄한 사진
- **picture** = (사진류 위조)
- **mask** = 종이/실리콘 마스크
- **display** = 휴대폰·태블릿 화면에 띄운 얼굴

뒤 4개는 전부 **위조(spoof)** 라서 막아야 해요.

---

## 2. 어떻게 판별하나요?

핵심 비결은 **카메라가 2개**라는 거예요:
- **RGB 카메라** = 우리 눈에 보이는 일반 컬러 영상
- **IR 카메라** = 적외선 영상

적외선으로 보면 **진짜 피부**와 **종이·화면**이 완전히 다르게 보여요. 사람 눈으론 구별 어려운 정밀한 사진도, 적외선에선 "이건 종이/화면이네"가 티가 나요. 그래서 RGB + IR 두 개를 같이 보면 위조를 훨씬 잘 잡아요.

모델(인공지능)은 **진짜/위조 예시 사진을 잔뜩 보여주며 학습**시켜서, 새 얼굴이 들어오면 진짜인지 위조인지 맞히게 만든 거예요.

---

## 3. 지금까지 한 일과 결과

1. **데이터 정리/병합**: 기존 데이터와 신규 데이터를 같은 구조로 맞추고, 얼굴 크롭을 10% 여백으로 다시 잡아 합쳤어요. 현재 통합 데이터는 3,849세션 / 15,396장이에요.
2. **학습**: 집에 있는 그래픽카드(GPU) 노트북(서브노트북)에서 모델을 학습시켰어요. CPU보다 GPU가 훨씬 빨라서 실제 학습은 서브노트북에서 돌리는 게 맞아요.
3. **결과**: 최신 float TFLite 평가에서 **"진짜 vs 위조" 판별 오류는 0**으로 나왔어요(APCER/BPCER/ACER 0). 전체 5종 분류 정확도는 약 89.1%였고, IR 카메라 덕을 크게 봤어요.
4. **코드 개선 (2026-06-28)**: 버그 수정, 데이터 증강 추가, 학습 스케줄러 추가, 중복 코드 정리를 했어요. MobileNetV2 기반 새 학습 코드도 실행 준비가 됐어요.

⚠️ 단, 주의할 점:
- 데이터는 늘었지만 아직 **독립 현장 테스트**가 아니에요. 처음 보는 사람, 다른 조명, 다른 거리에서도 이만큼 잘 될지는 더 검증해야 해요. 지금 숫자는 "좋아 보이지만 과신은 금물" 단계예요.
- "위조다"는 잘 맞히는데, **print(인쇄물) 세부 분류 recall이 0.48로 낮아요.** 다만 이 경우도 다른 위조 종류로 틀리는 쪽이면 출입 통제의 live/spoof 판정에는 직접 문제가 작아요.

---

## 4. 속도 문제와 "양자화" 이야기

모델이 보드(출입통제기)에서 한 번 판별하는 데 **80~220밀리초** 걸려요. 좀 느려요. 더 빠르게 하려면 보드에 있는 **NPU**(인공지능 전용 가속 칩)를 써야 해요.

그런데 이 NPU는 **"int8"이라는 가벼운 형식**의 모델만 빠르게 돌릴 수 있어요. 우리 모델은 지금 **무거운(정밀한) 형식**이라, 이걸 가벼운 형식으로 **변환(양자화, quantization)** 해야 NPU를 쓸 수 있어요.

비유하면: 고화질 원본 동영상(무겁지만 정밀)을 **압축**해서 가벼운 파일로 바꾸는 것과 비슷해요. 잘 압축하면 화질 손해가 거의 없지만, 잘못하면 화면이 깨지죠.

---

## 5. 양자화는 왜 어려웠고 어떻게 해결했나요? (솔직한 현황)

**과거에는 변환 도구가 에러를 내며 멈추거나 모델이 망가졌지만, 최근 실험을 통해 도구 에러를 완벽히 해결하고 돌파구를 찾았습니다.**

* **변환기 오류 해결 완료**: 
  가중치 위치 버그와 컴파일러 타입 에러를 우회하는 설정(`fold_quantize=False` 적용 및 `module()` 명시적 추출)을 찾아내어, 이제 **파이토치(PyTorch) 환경에서도 정상적으로 오류 없이 양자화된 TFLite 파일로 변환하는 데 성공**했습니다.
* **MobileNetV3의 한계 (정확도 붕괴)**:
  도구 에러는 해결했지만, **기존 MobileNetV3 모델은 양자화(INT8)만 거치면 정확도가 41% 수준으로 급락하며 정상 작동하지 않았습니다.** (진짜 얼굴을 정상 예측하지 못하는 현상 발생)
  이는 MobileNetV3 고유의 **SE 블록(채널을 8~16개로 극단적으로 줄이는 구조)**과 **Hard-swish 활성화 함수**의 값이 너무 작아, 정수(INT8)로 변환할 때 값이 통째로 `0`으로 뭉개져 버려 모델이 제 기능을 잃기 때문입니다.
* **해결책 - MobileNetV2 채택**:
  반면, **MobileNetV2**는 양자화에 매우 친화적인 **ReLU6 활성화 함수**를 사용하고 채널이 뭉개지는 SE 블록이 없습니다. 실제로 가상 변환 테스트에서도 **MobileNetV2를 쓰면 양자화 변환이 깨끗하게 성공**하며, 붕괴 현상이 전혀 발생하지 않는다는 사실을 입증했습니다.

---

## 6. 그래서 최종 결정과 권장 방향은?

**"속도 최적화(NPU 가속)와 모델 크기 축소를 위해 백본을 MobileNetV2로 결정"**

* **MobileNetV2를 쓰는 것이 훨씬 빠릅니다 (NPU 가속)**
  * **V3 (Float, CPU 구동)**: 크기는 조금 작아도 NPU 가속을 못 받아 **80~220ms**로 매우 느립니다.
  * **V2 (INT8, NPU 구동)**: 체급이 커도 NPU 가속이 작동하여 **5~10ms 이하**로 압도적으로 빠르게 동작합니다.
  * 게다가 압축(양자화) 덕분에 최종 파일 용량도 V3 Float(5.0MB)보다 **V2 INT8(3.5MB)**이 더 가볍습니다.
* **모델 사이즈를 더 줄이는 비결 (Width Multiplier)**
  V2의 용량을 V3-Small 수준으로 줄이고 속도를 더 올리고 싶다면, 모델의 채널 배율(Alpha)을 `0.75`나 `0.5`로 깎으면 됩니다. 이 설정을 통해 정확도는 높게 유지하면서 모델 사이즈는 2.0MB 수준까지 더 줄일 수 있습니다.
* **파이프라인 이원화**
  현재 개발팀은 **파이토치(PyTorch)와 케라스(Keras/TensorFlow) 양쪽 파이프라인 모두에서 MobileNetV2 기반의 학습 및 양자화 변환 흐름을 완전히 확보**한 상태입니다.
